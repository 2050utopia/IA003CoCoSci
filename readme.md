## 2. 协作规则

旧的文档和issue就先不管了（有闲暇自然也可以去进行修改）。之后发布/修改code、issue，都要遵从本文档的规则。  

初期可能会不适，习惯之后会很舒服。

### 2.1 文件(夹)

**命名**

* 英文，哪怕拼音。
* 全小写，单词之间用英文半角符号`-`连接，例如`bp-idea`
* 不可以有空格` `或斜杠`/`、`\`

* 参考的是阳老师构建优雅的知识创造系统里的截图。个人认为这种是比较利于在大时间周期复用的范式。自己实践过程发现，①不用切换大小写，减少认知负荷；②采用空格或斜杠可能在局部情景例如此次BP很方便，但是未来在其它场景会有许多麻烦，例如需要调用命令行来处理/操作文件时……

**管理**

待补充。我们的项目比其他人的小很多，涉及文件少，因此这次不用太在意。

### 2.2 issue标题命名

统一采用属性tag+“|”+标题内容的方式。例如：

	清单|计算认知科学重要信息源
	疑惑|xxxxxxxxx（疑惑点）
	待选定|到底挑选哪个大牛来探究呢？

避免指向不明的标题，例如`讨论|关于xxxxxx`。

太泛的内容要拆解成十几字以内可以描述完的内容。

### 2.3 commit message 用语

此次不过细地规定。中英文均可，但要尽可能地简明，尽可能在十个字以内描述完。

### 2.4 文档写作

适用于任意文档，例如code、issue、wiki。

#### 2.4.1 文档内容的组成成分

至少包括：

* 开头的简要说明——这个文档都有些什么。
* 正文
* changelog

#### 2.4.2 文档的标题结构

**数字+空格+标题内容**。本文到处都是例子。到达了五级标题(`#####`)的层面则只用标题内容。

例如：

	# 大标题
	## 1. 二级标题
	### 1.1 三级标题
	#### 1.1.1 四级标题
	##### 五级标题

PS：前期不确定顺序或是后续是否还会从中间插入新版块的话，可以用`2.x.x`或`2..`来代替，觉得某一部分不会大改了再加上数字。

#### 2.4.3 内容撰写的细节

这是为了提高代码层面的可读性。这样不需要依赖渲染器，在任意地方看这些文本都能一目了然。

* 标题之后空一行起文。  

例如：

	#### 2.4.3 内容撰写的细节

	这是为了提高代码层面的可读性。这样不需要依赖渲染器，在任意地方看这些文本都能一目了然。

* 文段结束之后空一行再起标题。

例如：

	适用于任意文档，例如code、issue、wiki。
	
	#### 2.4.1 文档内容的组成成分

* 文段与文段间空一行。

例如：

	EXAMPLEEXAMPLEEXAMPLEEXAMPLEEXAMPLEEXAMPLE   
	EXAMPLEEXAMPLEEXAMPLEEXAMPLEEXAMPLEEXAMPLE  
	EXAMPLEEXAMPLEEXAMPLEEXAMPLEEXAMPLEEXAMPLE  
	
	EXAMPLEEXAMPLEEXAMPLEEXAMPLEEXAMPLEEXAMPLE  
	EXAMPLEEXAMPLEEXAMPLEEXAMPLEEXAMPLEEXAMPLE  
	EXAMPLEEXAMPLEEXAMPLEEXAMPLEEXAMPLEEXAMPLE

### 2.6 changelog撰写

* 在任何情形下（无论是标题还是文中提及）都是小写不空格：`changelog`。
* 时间：采用6+4位数字。例如`190101 1800`。
* 内容：谁做了什么。例如人名+动词+名词：`麦云飞创建文档`。
* 新增的changelog永远浮在最顶上，旧的changelog逐渐沉底。
